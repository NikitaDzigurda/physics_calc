# Physics Calculator — Описание архитектуры с точки зрения ООП и Qt

---

## 1. Общая идея и разделение на слои

Проект разбит на **два независимых слоя**:

```
┌─────────────────────────────────────────────────┐
│                  Слой UI (Qt)                   │
│              src/ui/MainWindow                  │
│   QMainWindow · QPlainTextEdit · QTreeWidget    │
│        Signals & Slots · Shortcuts              │
└──────────────────────┬──────────────────────────┘
                       │ использует
┌──────────────────────▼──────────────────────────┐
│              Слой ядра (чистый C++)             │
│  Exceptions · Dimensions · Quantity             │
│  UnitRegistry · Lexer · Parser · Calculator     │
│  Никакой зависимости от Qt                      │
└─────────────────────────────────────────────────┘
```

**Зачем два слоя?**
Ядро можно скомпилировать и протестировать без Qt — как отдельную библиотеку. UI знает о ядре, ядро не знает об UI. Это классический принцип **разделения ответственности** (Separation of Concerns) и **инверсии зависимостей** (DIP из SOLID).

## 2. Иерархия классов ядра

### 2.1 `Dimensions` — вектор физических размерностей

```
┌─────────────────────────────────────────┐
│              Dimensions                 │
│─────────────────────────────────────────│
│ - exps_ : array<int, 7>                 │
│   [M, L, T, I, Θ, N, J]                │
│─────────────────────────────────────────│
│ + operator+= (сложение показателей)     │
│ + operator-= (вычитание показателей)    │
│ + pow(int n)                            │
│ + toString() : string                   │
│ + isScalar() : bool                     │
│ + static Base(Axis) : Dimensions        │
└─────────────────────────────────────────┘
```

**ООП-аспекты:**
- **Инкапсуляция** — массив `exps_` скрыт, снаружи доступны только именованные методы. Прямой доступ к индексам не нужен — это деталь реализации.
- **Перегрузка операторов** — `+=`, `-=`, `+`, `-`, `==`, `!=` делают код выражений читаемым: `dims_масса + dims_скорость` выглядит как обычная алгебра.
- **Статический фабричный метод** `Base(Axis::M)` возвращает готовый объект «единичная масса», не перекладывая на клиента знание внутреннего устройства.
- **`enum class Axis`** — типобезопасное именование осей; исключает путаницу с «магическими числами» `0..6`.

**Как работает алгебра размерностей:**

```
скорость = метр / секунда
[L]  -  [T]  =  [L, T⁻¹]   →   { 0,1,-1,0,0,0,0 }

сила = масса × ускорение
[M] + [L,T⁻²] = [M,L,T⁻²] →   { 1,1,-2,0,0,0,0 }  →  "N"
```

### 2.2 `Quantity<T>` — физическая величина (шаблонный класс)

```
┌─────────────────────────────────────────┐
│          Quantity<T>                    │
│─────────────────────────────────────────│
│ - value_ : T         (числовое значение)│
│ - dims_  : Dimensions (размерность)     │
│─────────────────────────────────────────│
│ + operator+, -, *, /                    │
│ + operator*(T scalar)                   │
│ + pow(int n)                            │
│ + toString() : string                   │
└─────────────────────────────────────────┘
```

**ООП-аспекты:**
- **Шаблон (Template / Generic programming)** — `Quantity<double>`, `Quantity<float>` и т. д. Код написан один раз и работает для любого числового типа. В проекте используется алиас `using Q = Quantity<double>`.
- **Инвариант класса** — объект `Quantity` всегда содержит согласованную пару «значение + размерность». Нельзя изменить одно без другого.
- **Перегрузка операторов** с проверкой размерностей:
  - `+` / `-` → бросают `DimensionMismatch` если размерности не совпадают.
  - `*` / `/` → складывают/вычитают показатели размерностей.
- **Принцип наименьшего удивления** — `Quantity * Quantity`, `Quantity * double` и `double * Quantity` все работают интуитивно, как в математике.

**Пример: умножение двух величин**

```cpp
Q mass(70.0, Dimensions::Base(M));        // 70 kg
Q accel(9.8, Dims({0,1,-2,0,0,0,0}));     // 9.8 m/s²
Q force = mass * accel;                    // 686 N
// Внутри: value = 70*9.8, dims = [1,0,0] + [0,1,-2] = [1,1,-2] → "N"
```

### 2.3 Иерархия исключений

```
std::exception
 └── std::runtime_error
      └── CalcError              ← базовый класс для всех ошибок проекта
           ├── DimensionMismatch ← несовместимые размерности
           ├── ParseError        ← ошибка синтаксиса (+ позиция символа)
           ├── EvalError         ← ошибка вычисления (деление на ноль, …)
           └── UnknownIdentifier ← неизвестный символ (+ имя символа)
```

**ООП-аспекты:**
- **Наследование** — вся иерархия строится через публичное наследование `std::runtime_error`. Это соответствует принципу подстановки Лисков (LSP): везде, где ожидается `CalcError`, можно подставить любой дочерний класс.
- **Полиморфизм** — `Calculator::evaluate` ловит `const CalcError&` и обрабатывает любую ошибку проекта одним `catch`.
- **Специализация данных** — `ParseError` хранит `position` (для подсветки ошибки), `UnknownIdentifier` хранит `identifier` (для подстановки в сообщение).

### 2.4 `UnitRegistry` — реестр единиц и констант

```
┌───────────────────────────────────────────────┐
│                 UnitRegistry                  │
│───────────────────────────────────────────────│
│ - table_   : unordered_map<string, Q>         │
│ - sections_: vector<Section>                  │
│   Section { title; entries: vector<Entry> }   │
│   Entry   { symbol; description; Q value }    │
│───────────────────────────────────────────────│
│ + has(name) : bool                            │
│ + get(name) : Q     // бросает UnknownIdent.  │
│ + sections(): const vector<Section>&          │
└───────────────────────────────────────────────┘
```

**ООП-аспекты:**
- **Инкапсуляция** — приватный метод `addUnit()` — единственное место вставки. Конструктор вызывает его 60+ раз, строя каталог последовательно и читаемо.
- **Агрегация** — `UnitRegistry` агрегирует вложенные структуры `Section` и `Entry`. Это пример отношения «целое — часть», где части не существуют без реестра.
- **Двойное представление** — `table_` для O(1)-поиска по имени, `sections_` для отображения в дереве UI. Оба синхронизируются при каждом `addUnit`.
- **Принцип открытости/закрытости (OCP)** — чтобы добавить новую единицу, достаточно одной строки в конструкторе; всё остальное не меняется.

### 2.5 `Lexer` — лексический анализатор

```
┌───────────────────────────────────────────────┐
│                   Lexer                       │
│───────────────────────────────────────────────│
│ - src_ : string                               │
│ - i_   : size_t   (текущая позиция)           │
│───────────────────────────────────────────────│
│ + tokenize() : vector<Token>                  │
│ - readNumber() : Token                        │
│ - readIdent()  : Token                        │
└───────────────────────────────────────────────┘

struct Token {
    Tok    kind;    // Number, Ident, Plus, Minus, Star, Slash, Caret, ...
    string text;
    double number;
    size_t pos;
}
```

**ООП-аспекты:**
- **Инкапсуляция** — приватные вспомогательные методы `readNumber` / `readIdent` скрывают детали разбора. Наружу — один метод `tokenize()`.
- **Единственная ответственность (SRP)** — Lexer знает только как превратить строку в токены. Он не знает ни о физике, ни о вычислении.
- **Defensive programming** — числа читаются через `istringstream` с классической C-локалью (`imbue(locale::classic())`), что решает проблему запятой в русской системной локали.

### 2.6 `Parser` — синтаксический анализатор и вычислитель

```
┌────────────────────────────────────────────────────────────┐
│                        Parser                              │
│────────────────────────────────────────────────────────────│
│ - tokens_  : vector<Token>                                 │
│ - i_       : size_t                                        │
│ - calc_    : Calculator&   ← ссылка для lookup/assign      │
│────────────────────────────────────────────────────────────│
│ + parse() : Result { value; wasAssignment; assignedName }  │
│ - parseExpr()    expr   := term   (('+' | '-') term)*      │
│ - parseTerm()    term   := factor (('*'|'/') factor)*      │
│ - parseFactor()  factor := unary  ('^' factor)?            │
│ - parseUnary()   unary  := ('+' | '-') unary | atom        │
│ - parseAtom()    atom   := NUMBER | IDENT | '(' expr ')'   │
└────────────────────────────────────────────────────────────┘
```

**ООП-аспекты:**
- **Паттерн «Рекурсивный спуск»** — каждое грамматическое правило реализовано как отдельный приватный метод. Методы вызывают друг друга в иерархии, что напрямую отражает грамматику.
- **Зависимость через ссылку** — Parser получает `Calculator&` в конструкторе. Он не создаёт калькулятор сам. Это **инъекция зависимости** (DI): зависимость передаётся снаружи, а не создаётся внутри.
- **«Умное» умножение** — в `parseTerm` если после `factor` идёт другой `atom` без оператора (`9.8 m/s^2`), это неявно трактуется как умножение. Правило добавлено в одном месте, не меняя остальную грамматику.

---

### 2.7 `Calculator` — фасад и хранитель состояния

```
┌────────────────────────────────────────────────┐
│                  Calculator                    │
│────────────────────────────────────────────────│
│ - units_   : UnitRegistry                      │
│ - vars_    : unordered_map<string, Q>          │
│ - history_ : vector<HistoryEntry>              │
│────────────────────────────────────────────────│
│ + evaluate(input) : HistoryEntry               │
│ + lookup(name)    : Q      ← vars_ → units_   │
│ + assign(name, Q)                              │
│ + clearVariables()                             │
│ + clearHistory()                               │
└────────────────────────────────────────────────┘

struct HistoryEntry { input; output; note; isError }
```

**ООП-аспекты:**
- **Паттерн «Фасад»** — одна точка входа для всего вычислительного стека. UI вызывает только `evaluate(string)` и не знает ни про Lexer, ни про Parser.
- **Инкапсуляция состояния** — переменные и история хранятся внутри объекта. Один `Calculator` на окно = одна сессия с изолированным контекстом.
- **Приоритет поиска** в `lookup`: сначала `vars_`, потом `units_`. Это реализует механизм затенения встроенных имён пользовательскими переменными — в одном методе, без условной логики в парсере.
- **Обработка ошибок** — `evaluate` ловит все исключения семейства `CalcError` и превращает их в поле `isError` объекта `HistoryEntry`. UI никогда не видит необработанных исключений из ядра.

## 3. Схема взаимодействия объектов (поток одного вычисления)

```
MainWindow
    │
    │  evaluate("force = mass * accel")
    ▼
Calculator::evaluate(input)
    │
    ├─► Lexer(input).tokenize()
    │       → [Ident"force", Assign, Ident"mass", Star, Ident"accel", End]
    │
    ├─► Parser(tokens, *this).parse()
    │       │
    │       ├─ parseFactor("mass")   → lookup("mass")  → vars_["mass"] = 70 kg
    │       ├─ parseFactor("accel")  → lookup("accel") → vars_["accel"] = 9.8 m/s²
    │       ├─ 70 kg * 9.8 m/s²     → Q(686, N)
    │       └─ assign("force", Q(686, N))
    │
    └─► HistoryEntry { "force = mass * accel", "force = 686 N", false }
    │
    ▼
MainWindow::appendHistory(entry)
MainWindow::refreshVariables()
```

## 4. Qt — использованные паттерны и классы

### 4.1 Структура главного окна (`MainWindow`)

```
MainWindow : QMainWindow
│
├── QToolBar                        ← Clear input / Clear history / Reset / Help
│
└── QSplitter (вертикальный)
     ├── inputBox (QWidget)
     │    ├── QLabel                ← подпись
```

### 4.2 Сигналы и слоты — центральный механизм Qt

```
┌──────────────────────────────────────────────────────────────┐
│              СИГНАЛЫ              →         СЛОТЫ            │
├──────────────────────────────────────────────────────────────┤
│ evalBtn_::clicked                 → MainWindow::onEvaluate   │
│ QShortcut(⌘+Enter)::activated     → MainWindow::onEvaluate   │
│ clearHist::triggered              → MainWindow::onClearHist  │
│ clearVars::triggered              → MainWindow::onClearVars  │
│ reference_::itemDoubleClicked     → MainWindow::onRefClick   │
│ help::triggered                   → MainWindow::onShowHelp   │
└──────────────────────────────────────────────────────────────┘
```

**Qt-аспекты:**
- **Сигналы и слоты** — механизм Qt для связи между объектами без прямой зависимости. Кнопка не знает, кто её слушает; слот не знает, что его вызвало.
- **`Q_OBJECT`** — макрос в заголовке `MainWindow`, обязательный для классов с сигналами/слотами. Он даёт Qt инструмент MOC (Meta-Object Compiler) необходимую метаинформацию.
- **Лямбда в `connect`** — современный стиль (Qt5+):
  ```cpp
  connect(clearInput, &QAction::triggered,
      this, [this] { input_->clear(); input_->setFocus(); });
  ```
  Позволяет задавать слот-коллбэк прямо в месте подключения, не создавая лишних методов.
- **`QShortcut`** — привязка горячих клавиш не к конкретному виджету, а к окну. `Qt::CTRL | Qt::Key_Return` на macOS автоматически преобразуется в `⌘+Return`.

### 4.3 MOC — система мета-объектов Qt

Qt добавляет поверх стандартного C++ слой метаинформации о классах:

```
MainWindow.h  ──► MOC ──► mocs_compilation.cpp
                          (генерируется автоматически при сборке)
```

MOC обрабатывает:
- **`Q_OBJECT`** — включает систему сигнал/слот, RTTI Qt, `tr()` для локализации.
- **`signals:`** — объявление сигналов (генерирует код отправки).
- **`slots:`** — объявление слотов (регистрирует их в мета-таблице).

В `CMakeLists.txt` включается командой:
```cmake
set(CMAKE_AUTOMOC ON)
```

### 4.6 Локаль — системная проблема и её решение

**Проблема:** `QApplication` на старте подхватывает системную локаль. На машинах с русской локалью `LC_NUMERIC` переключается на «запятая как разделитель» (`9,8`). Стандартная `std::strtod("9.8")` прочитает только `9` и остановится.

**Решение — два уровня:**

```
main.cpp:
    QApplication app(argc, argv);
    std::setlocale(LC_NUMERIC, "C");   ← принудительно вернуть точку

Lexer.cpp:
    std::istringstream iss(t.text);
    iss.imbue(std::locale::classic()); ← независимо от глобальной локали
    iss >> t.number;

Quantity.h:
    os.imbue(std::locale::classic());  ← вывод тоже с точкой
```

## 5. SOLID-принципы в проекте

| Принцип | Где реализован |
|---------|---------------|
| **S** — Single Responsibility | `Lexer` только токенизирует; `Parser` только разбирает; `Calculator` только хранит состояние и запускает pipeline |
| **O** — Open/Closed | Добавить единицу — одна строка в `UnitRegistry`; добавить операцию — одно правило в `Parser` |
| **L** — Liskov Substitution | `DimensionMismatch`, `ParseError`, `EvalError` подставляются везде, где ожидается `CalcError` |
| **I** — Interface Segregation | UI видит только `Calculator` (фасад); не знает о `Lexer`, `Parser`, `Dimensions` |
| **D** — Dependency Inversion | `Parser` зависит от абстракции `Calculator&` (ссылки), а не создаёт его сам |

## 6. Шаблоны проектирования (Design Patterns)

| Паттерн | Класс | Суть |
|---------|-------|------|
| **Фасад** (Facade) | `Calculator` | Скрывает всю цепочку Lexer→Parser за методом `evaluate(string)` |
| **Шаблонный метод** (Template Method) | `Quantity<T>` | Алгоритм арифметики фиксирован, тип данных подставляется снаружи |
| **Стратегия поиска** | `Calculator::lookup` | Порядок поиска: vars → units — в одном методе, без условий снаружи |
| **Компоновщик** (Composite) | `QTreeWidget` | Секция содержит элементы, оба являются `QTreeWidgetItem` |
| **Наблюдатель** (Observer) | Signals & Slots | `QPushButton` излучает сигнал, `MainWindow` реагирует через слот |
| **Фабричный метод** | `Dimensions::Base(Axis)` | Создание размерности по имени оси без знания внутреннего формата |

## 8. Сборочная система (CMake + Qt6)

```cmake
find_package(Qt6 6.2 REQUIRED COMPONENTS Widgets)

set(CMAKE_AUTOMOC ON)   ← запускает MOC для Q_OBJECT классов
set(CMAKE_AUTORCC ON)   ← обрабатывает .qrc файлы ресурсов
set(CMAKE_AUTOUIC ON)   ← обрабатывает .ui файлы (в проекте не используются)

qt_add_executable(physics_calc ...)
target_link_libraries(physics_calc PRIVATE Qt6::Widgets)
```

**Ключевые моменты:**
- `qt_add_executable` вместо `add_executable` — создаёт `.app`-бандл на macOS автоматически.
- `MACOSX_BUNDLE TRUE` — стандартная macOS-обёртка (папка `physics_calc.app`).
- `PRIVATE` в `target_link_libraries` — Qt-зависимость не экспортируется потребителям этой цели.
